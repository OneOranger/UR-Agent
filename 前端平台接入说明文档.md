# UR Agent 前端平台接入说明文档

## 1. 文档目标

本文档用于说明当前 **UR Agent 后端** 如何接入前端平台，包括：

* 页面结构设计
* 前后端交互流程
* 关键接口说明
* 状态管理建议
* 数据展示建议
* 审核流（HITL）接入方式
* 报告展示方式
* Prompt 管理后续接入建议

目标是让前端团队可以直接基于当前后端接口开始搭建平台原型。

---

## 2. 当前后端能力概览

当前后端已经具备以下核心能力：

1. 创建研究任务
2. 基于 LangGraph 执行完整研究流程
3. 在审核节点中断并等待人工审批
4. 恢复任务继续执行
5. 查询完整 run 详情
6. 返回研究中间产物与最终报告
7. 返回审批历史、trace、评估结果
8. 导出 Markdown 报告并保存路径
9. Prompt 资产统一管理

当前完整研究链路：

```text
evidence -> themes -> insights -> personas -> journey_maps -> recommendations -> report
```

---

## 3. 前端平台建议页面结构

建议前端平台第一期至少做 4 个核心页面。

### 3.1 新建研究任务页

路径建议：

```text
/research/new
```

功能：

* 输入研究需求
* 触发创建研究任务
* 展示初始执行状态
* 保存 `run_id` 与 `thread_id`

页面字段建议：

* 研究需求输入框（多行文本）
* 提交按钮
* 创建后跳转到任务详情页

---

### 3.2 研究任务详情页

路径建议：

```text
/research/:runId
```

功能：

* 查看任务当前状态
* 查看所有中间产物
* 查看最终报告
* 查看评估结果
* 查看 trace
* 如果任务中断，可跳转到审核页

建议以 Tab 形式展示以下模块：

* 概览
* 证据 Evidence
* 主题 Themes
* 洞察 Insights
* 画像 Personas
* 旅程 Journey Maps
* 建议 Recommendations
* 报告 Report
* 审批 Approvals
* Trace
* 评估 Evals

---

### 3.3 审核页

路径建议：

```text
/research/:runId/review
```

功能：

* 显示当前待审核任务的报告摘要
* 显示审批类型和说明
* 通过 / 驳回
* 输入审批备注

页面字段建议：

* 当前审核类型（如 final_report）
* 执行摘要
* 通过按钮
* 驳回按钮
* 备注输入框

---

### 3.4 报告展示页

路径建议：

```text
/research/:runId/report
```

功能：

* 只展示结构化报告
* 适合作为阅读页或导出前预览页

建议展示字段：

* 标题
* 背景
* 研究目标
* 执行摘要
* 关键发现
* 主题总结
* 用户画像总结
* 用户旅程总结
* 建议
* 方法
* 局限性
* 下一步

> 注意：前端报告展示应直接使用结构化 `report` 字段，不要依赖 Markdown 文件作为主数据源。

## 4. 前后端关键接口

当前前端接入最核心的 3 个接口如下。

---

### 4.1 创建研究任务

```http
POST /research/run
```

请求体：

```json
{
  "user_request": "为一个面向东京年轻白领的健身 App 做用户研究"
}
```

返回核心字段：

```json
{
  "thread_id": "...",
  "status": "interrupted 或 completed",
  "interrupts": [...],
  "result": {
    "run_id": "...",
    "thread_id": "...",
    "current_stage": "...",
    "research_plan": {
      "markdown_report_path": "..."
    }
  }
}
```

前端必须保存：

* `run_id`
* `thread_id`

说明：

* `run_id` 用于查询任务详情
* `thread_id` 用于恢复被中断的任务

---

### 4.2 恢复被中断任务（审批）

```http
POST /research/resume
```

请求体：

```json
{
  "thread_id": "xxx",
  "decision": "approved",
  "comment": "通过"
}
```

或驳回：

```json
{
  "thread_id": "xxx",
  "decision": "rejected",
  "comment": "请补充更多证据"
}
```

返回：

* 最新任务执行结果
* 最新 `result`
* 最新状态

说明：

* 恢复时只需要 `thread_id`
* 恢复完成后，前端建议重新调用详情接口刷新页面

---

### 4.3 查询任务详情

```http
GET /research/run/{run_id}
```

返回结构目前包含：

* `run`
* `evidence_items`
* `themes`
* `insights`
* `personas`
* `journey_maps`
* `recommendations`
* `report`
* `approvals`
* `traces`
* `eval_summary`
* `eval_records`

这是前端详情页的主数据源。

## 5. 前端状态管理建议

前端建议统一保存以下核心状态。

```ts
type ResearchTaskState = {
  runId: string
  threadId: string
  status: 'running' | 'interrupted' | 'completed'
  currentStage?: string
  detail?: any
}
```

建议逻辑：

### 创建任务后

保存：

* `runId = result.run_id`
* `threadId = thread_id`
* `status = interrupted/completed`

### 如果任务中断

* 跳转审核页
* 或在详情页显示“待审批”横幅

### 审批完成后

* 再调用 `GET /research/run/{run_id}` 刷新详情
* 用最新详情覆盖前端状态

## 7. 审核流（HITL）接入说明

当前审核流是整个前后端交互里最重要的一块。

### 7.1 后端工作方式

后端在 `request_final_approval` 节点会中断图执行，等待人工审批。

### 7.2 前端如何识别中断

当前端调用：

```http
POST /research/run
```

如果返回：

```json
{
  "status": "interrupted"
}
```

说明当前任务停在审批节点。

### 7.3 前端如何处理

前端应：

1. 展示审核页或审核弹窗
2. 显示当前 report 核心摘要
3. 让用户填写评论
4. 调用 `/research/resume`

## 8. 前端技术选型建议

如果你现在准备单独做平台前端，建议采用：

### 推荐组合

* React + Vite
* TypeScript
* Ant Design 或 shadcn/ui
* React Query 管理接口请求
* Axios 发送请求

### 原因

* 开发速度快
* 类型清晰
* 容易做表格 / Tabs / 审核页
* 和当前 JSON 接口非常匹配

## 12. 当前后端与前端的最佳连接方式

最重要的 3 条规则：

### 规则 1

前端主数据源永远是：

```http
GET /research/run/{run_id}
```

### 规则 2

恢复流程永远用：

* `thread_id`

不是 `run_id`

### 规则 3

报告展示优先使用结构化 `report` 数据，不依赖 Markdown 解析。

Markdown 文件更适合作为导出产物，而不是前端主展示数据。
